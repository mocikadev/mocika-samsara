集成方式：MCP 客户端
集成成本：1-2 周

## 三、分布式 AGENTS.md 网络组织模式

### 模式对比

| 模式 | 结构 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **嵌套** | 全局 → 项目 → 目录 | 简单、向后兼容 | 无法处理大型项目 | 小型项目、单仓库 |
| **分层** | Tier 1 (路由) + Tier 2 (子系统) | 支持大型项目、token 高效 | 需要手动维护 | 中型项目、多子系统 |
| **中央仓库** | 中央规则 → CI 生成 → 各项目 | 跨组织统一、易于维护 | 灵活性低 | 大型组织、多项目 |
| **混合** | 嵌套 + 分层 + 中央仓库 | 最大灵活性、支持所有场景 | 复杂度高 | 企业级、多层级 |

### 推荐选择

**混合模式（嵌套 + 分层 + 中央仓库）**

```
全局层：~/.agents/AGENTS.md
  ├─ 个人偏好
  ├─ 多仓库上下文
  └─ 工作流规则

项目层：project-root/AGENTS.md
  ├─ 项目范围规则
  ├─ 构建/测试/lint 命令
  └─ 命名规范

子系统层：project-root/.agents/*.md
  ├─ API 子系统规则
  ├─ 数据库层规则
  └─ Web 前端规则

中央仓库：agent-standards/ repo
  ├─ 跨组织规则
  ├─ 共享 skills
  └─ 模板库
```

**优势**：
- 支持个人、项目、组织三个层级
- 后面的文件覆盖前面的（优先级清晰）
- 与 OpenAI、Datadog 等大型组织实践对齐
- 向后兼容现有 AGENTS.md 标准

## 四、Samsara Layer C 实现方案

### 三阶段实现计划

| 阶段 | 目标 | 工作量 | 时间 | 交付物 |
|------|------|--------|------|--------|
| **Phase 1** | 代码图谱集成 | 中等 | 2-3 周 | `samsara scan` 命令 |
| **Phase 2** | AGENTS.md 生成 | 中等 | 2-3 周 | `samsara generate-agents-md` 命令 |
| **Phase 3** | 分布式网络 | 低 | 1-2 周 | `samsara sync-network` 命令 |

### 推荐工具组合

| 组件 | 推荐工具 | 备选方案 | 集成方式 |
|------|---------|---------|---------|
| 代码图谱 | CodeMap | StakGraph / Codebase-Memory | MCP 客户端 |
| AGENTS.md 生成 | agentmd (CLI) | agents-md-generator (MCP) | 子进程调用 |
| 分布式管理 | 自定义（git + INDEX.md） | Nx monorepo 工具 | git hooks |
| 增量更新 | 基于文件 hash | 基于 git diff | 后台守护进程 |

### 与 Samsara 现有设计的兼容性

✅ **完全兼容**：
- MCP 标准与跨工具兼容目标一致
- 增量扫描与"被动积累"模式对齐
- INDEX.md 全量重建方案可扩展
- 分层 AGENTS.md 与分层知识结构对齐

⚠️ **需要注意**：
- 代码图谱存储位置：`~/.agents/code-graphs/`
- 缓存管理：定期清理过期数据
- 权限管理：多项目访问控制

## 五、关键决策（P-07 ~ P-09）

### P-07：是否实现 Layer C？

| 选项 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **实现** | 自动化、支持大型库、符合用户期望 | 需要集成外部工具 | ✅ 推荐 |
| **不实现** | 减少工作量 | 无法自动化、用户体验差 | ❌ 不推荐 |

**理由**：
1. 现有工具已成熟（生产级）
2. MCP 标准化使集成成本低
3. 与 Samsara 核心价值对齐
4. 用户期望（AGENTS.md 已成行业标准）

---

### P-08：AGENTS.md 质量保证？

| 选项 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **生成 + 评分 + 漂移检测** | 三层保证、自动检测、CI 集成 | 需要集成 agentmd | ✅ 推荐 |
| **仅生成** | 实现简单 | 无法保证质量 | ❌ 不推荐 |
| **完全手动** | 完全控制 | 不符合自动化目标 | ❌ 不推荐 |

**质量目标**：
- 总分 ≥ 80/100
- 新鲜度 ≥ 15/20
- 月度漂移检测

---

### P-09：分布式网络组织模式？

| 选项 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **嵌套 + 分层 + 中央仓库** | 最大灵活性、支持所有场景 | 复杂度高 | ✅ 推荐 |
| **仅嵌套** | 简单 | 无法处理大型项目 | ❌ 不推荐 |
| **仅中央仓库** | 易于维护 | 灵活性低 | ❌ 不推荐 |

**理由**：
1. 与大型组织实践对齐
2. 支持个人、项目、组织三层级
3. 最大灵活性
4. 向后兼容
